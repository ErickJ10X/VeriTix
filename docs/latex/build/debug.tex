\subsection{Plataformas de ticketing}\label{plataformas-de-ticketing}}

Empresas que comercializan entradas de terceros y operan con alto

concurrencia.

\hypertarget{operadores-de-acceso-y-validaciuxf3n}{%
\subsection{Operadores de acceso y
validación}\label{operadores-de-acceso-y-validaciuxf3n}}

Equipos y empresas que gestionan el acceso físico al recinto y requieren
validación rápida, trazable y resistente a intentos de reutilización de
entradas.

\hypertarget{ejemplo-de-estructura-organizativa-del-sector}{%
\section{Ejemplo de estructura organizativa del
sector}\label{ejemplo-de-estructura-organizativa-del-sector}}

Una ticketera digital de tamaño medio suele organizarse en:

\begin{itemize}
\tightlist
\item
  \textbf{Dirección:} estrategia y priorización de negocio.
\item
  \textbf{Operaciones:} relación con organizadores y operación de
  eventos.
\item
  \textbf{Tecnología:} backend, frontend, base de datos, seguridad y
  despliegue.
\item
  \textbf{Comercial/marketing:} captación de organizadores y promoción.
\item
  \textbf{Atención al cliente:} incidencias de compra y acceso.
\item
  \textbf{Validación en campo:} control de acceso durante el evento.
\end{itemize}

\hypertarget{necesidades-que-cubre-veritix}{%
\section{Necesidades que cubre
VeriTix}\label{necesidades-que-cubre-veritix}}

VeriTix aborda necesidades reales del sector con alcance verificable en
el repositorio:

\begin{itemize}
\tightlist
\item
  \textbf{Centralización operativa:} gestión de eventos, catálogos y
  usuarios desde una API unificada.
\item
  \textbf{Control de stock y sobreventa:} compra con transacciones
  atómicas en PostgreSQL/Prisma.
\item
  \textbf{Trazabilidad del acceso:} tickets con hash único y registro de
  validación (\texttt{validatedAt}, \texttt{validatedBy}).
\item
  \textbf{Automatización del flujo de cobro:} integración con Stripe
  Checkout + webhooks.
\item
  \textbf{Notificación transaccional:} envío de emails de verificación y
  confirmación mediante Resend y BullMQ.
\end{itemize}

\hypertarget{descripciuxf3n-del-proyecto}{%
\section{Descripción del proyecto}\label{descripciuxf3n-del-proyecto}}

VeriTix es un monorepo con dos aplicaciones desacopladas:

\begin{itemize}
\tightlist
\item
  \textbf{Backend:} NestJS 11 + TypeScript + Prisma 7 + PostgreSQL +
  Redis.
\item
  \textbf{Frontend:} Nuxt 4 + Vue 3 + Nuxt UI + Tailwind 4.
\end{itemize}

La autenticación se basa en JWT (access token de corta vida y refresh
token en cookie HTTP-only con rotación). El dominio contempla roles
\texttt{BUYER}, \texttt{CREATOR}, \texttt{VALIDATOR} y \texttt{ADMIN}.

\hypertarget{justificaciuxf3n-del-tipo-de-soluciuxf3n}{%
\section{Justificación del tipo de
solución}\label{justificaciuxf3n-del-tipo-de-soluciuxf3n}}

Se selecciona arquitectura web desacoplada (API REST + cliente web
SSR/SPA) por:

\begin{itemize}
\tightlist
\item
  \textbf{Escalabilidad operativa:} separación de backend y frontend
  para evolucionar cada capa.
\item
  \textbf{Mantenibilidad:} módulos de dominio en NestJS y tipado fuerte
  extremo a extremo.
\item
  \textbf{Portabilidad:} la API puede reutilizarse por otros clientes
  (ej. app móvil).
\item
  \textbf{Accesibilidad:} acceso multiplataforma sin instalación de
  software cliente nativo.
\end{itemize}

\hypertarget{modelo-de-datos-y-constraints}{%
\subsubsection{Modelo de datos y
constraints}\label{modelo-de-datos-y-constraints}}

El esquema Prisma (\texttt{backend/prisma/schema.prisma}) define como
núcleo transaccional: \texttt{User}, \texttt{RefreshToken},
\texttt{Event}, \texttt{TicketType}, \texttt{Order}, \texttt{OrderItem},
\texttt{Ticket}, \texttt{Payment}; y como catálogos/soporte:
\texttt{Venue}, \texttt{Artist}, \texttt{Genre}, \texttt{ConcertFormat},
\texttt{EventArtist}.

\textbf{Relaciones principales verificables}

\begin{itemize}
\tightlist
\item
  \texttt{User\ (creator)} 1:N \texttt{Event}.
\item
  \texttt{Event} 1:N \texttt{TicketType}, \texttt{Order},
  \texttt{Ticket}.
\item
  \texttt{Order} 1:N \texttt{OrderItem}, 1:N \texttt{Payment}, 1:N
  \texttt{Ticket}.
\item
  \texttt{TicketType} 1:N \texttt{OrderItem} y 1:N \texttt{Ticket}.
\item
  \texttt{User\ (buyer)} 1:N \texttt{Order} y 1:N \texttt{Ticket}.
\item
  \texttt{User\ (validator)} 1:N \texttt{Ticket} validado (relación
  opcional \texttt{validatedBy}).
\item
  \texttt{Event} N:M \texttt{Artist} mediante \texttt{EventArtist} (con
  \texttt{role} y \texttt{performanceOrder}).
\item
  \texttt{Event} N:M \texttt{Genre}; \texttt{Artist} N:M \texttt{Genre}.
\end{itemize}

\textbf{Constraints e índices relevantes (extracto)}

\begin{itemize}
\tightlist
\item
  Unicidad: \texttt{users.email}, \texttt{users.phone},
  \texttt{tickets.hash}, \texttt{venues.slug}, \texttt{artists.slug},
  \texttt{genres.name/slug}, \texttt{concert\_formats.name/slug}, y
  \texttt{event\_artists(event\_id,\ artist\_id)}.
\item
  Integridad referencial con FK: en todas las relaciones principales
  (\texttt{event\_id}, \texttt{buyer\_id}, \texttt{order\_id}, etc.).
\item
  Cascadas explícitas (\texttt{onDelete:\ Cascade}) en relaciones
  críticas de limpieza: \texttt{RefreshToken\ -\textgreater{}\ User},
  \texttt{EventArtist\ -\textgreater{}\ Event},
  \texttt{TicketType\ -\textgreater{}\ Event},
  \texttt{OrderItem\ -\textgreater{}\ Order},
  \texttt{Ticket\ -\textgreater{}\ Event}.
\item
  Índices de consulta: por ejemplo \texttt{events(status,\ eventDate)},
  \texttt{orders(buyerId,\ createdAt)},
  \texttt{orders(eventId,\ status)}, \texttt{tickets(buyerId,\ status)}
  y \texttt{tickets(eventId,\ status)}.
\item
  Estados de dominio mediante enums (\texttt{Role},
  \texttt{EventStatus}, \texttt{OrderStatus}, \texttt{TicketStatus},
  \texttt{PaymentStatus}) para reducir ambigüedad semántica.
\end{itemize}

\textbf{Limitaciones actuales del esquema (a considerar)}

\begin{itemize}
\tightlist
\item
  No se observan \texttt{CHECK} constraints a nivel BD para invariantes
  como \texttt{availableQuantity\ \textgreater{}=\ 0},
  \texttt{totalQuantity\ \textgreater{}=\ availableQuantity} o
  coherencia temporal
  (\texttt{saleStartDate\ \textless{}=\ saleEndDate}); estas reglas
  dependen de validaciones de aplicación.
\item
  \texttt{currency} se modela como \texttt{String} (no enum/referencia),
  lo que permite mayor flexibilidad pero requiere disciplina en
  validación de códigos ISO.
\item
  En algunas FKs no se declara \texttt{onDelete} explícito (se delega al
  comportamiento por defecto de Prisma/PostgreSQL), por lo que conviene
  revisar políticas de borrado para operación productiva.
\end{itemize}

verificable)}\label{ejecuciuxf3n-del-proyecto-estado-verificable}}

\textbf{Ficheros de configuración relevantes:}

\begin{itemize}
\tightlist
\item
  \textbf{\texttt{backend/.env.example}:} variables necesarias del
  backend (\texttt{DATABASE\_URL}, secrets JWT, Stripe, Resend, Redis,
  CORS, etc.).
\item
  \textbf{\texttt{backend/src/config/env.validation.ts}:} validación Joi
  de variables críticas en arranque.
\item
  \textbf{\texttt{backend/prisma/schema.prisma}:} modelo de datos y
  restricciones.
\item
  \textbf{\texttt{backend/docker-compose.yml} y
  \texttt{frontend/docker-compose.yml}:} configuración de contenedores
  por paquete. (No existe un \texttt{docker-compose.yml} único en la
  raíz).
\end{itemize}

\textbf{Aspectos técnicos confirmados:}

\begin{itemize}
\tightlist
\item
  API con prefijo global \texttt{/api/v1}.
\item
  Documentación Swagger activa en \texttt{/docs}.
\item
  Webhook de Stripe en \texttt{/api/v1/webhooks/stripe} con
  \texttt{rawBody} habilitado para firma.
\item
  Seguridad con \texttt{helmet}, \texttt{cookie-parser}, validación
  global y guards.
\end{itemize}

\hypertarget{incidencias}{%
\section{Incidencias}\label{incidencias}}

\hypertarget{protocolo-de-resoluciuxf3n}{%
\subsection{Protocolo de resolución}\label{protocolo-de-resoluciuxf3n}}

\hypertarget{recopilaciuxf3n-de-informaciuxf3n}{%
\subsubsection{Recopilación de
información}\label{recopilaciuxf3n-de-informaciuxf3n}}

Para cada incidencia se debe registrar: contexto funcional, pasos de
reproducción, rol afectado, hora, entorno (local/staging/prod), trazas
relevantes y severidad.

\hypertarget{diagnuxf3stico-y-soluciuxf3n}{%
\subsubsection{Diagnóstico y
solución}\label{diagnuxf3stico-y-soluciuxf3n}}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Clasificar incidencia (funcional, seguridad, rendimiento, datos, UX).
\item
  Reproducir en entorno controlado.
\item
  Aplicar corrección mínima verificable.
\item
  Ejecutar pruebas de regresión del módulo impactado.
\end{enumerate}

\hypertarget{registro}{%
\subsubsection{Registro}\label{registro}}

Las incidencias deben documentarse en GitHub Issues/PR con: descripción,
evidencia, raíz del problema, cambio aplicado, pruebas asociadas y
estado de cierre.

\hypertarget{documento-de-pruebas-a-realizar}{%
\section{Documento de pruebas a
realizar}\label{documento-de-pruebas-a-realizar}}

La estrategia de pruebas se estructura por niveles y distingue entre
evidencia existente en repositorio y cobertura aún pendiente de
integración funcional.

\hypertarget{trazabilidad-con-contratos-api-y-modelo-de-datos}{%
\subsection{Trazabilidad con contratos API y modelo de
datos}\label{trazabilidad-con-contratos-api-y-modelo-de-datos}}

La identificación de endpoints reales, constraints de datos y resumen
entidad-relación se documenta en
\texttt{docs/latex/src/02-identificacion-necesidades.md} (subsecciones:
\emph{API real del sistema}, \emph{Modelo de datos y constraints} y
\emph{Diagrama entidad-relación (resumen)}). Esta referencia permite
alinear criterios de prueba con contratos efectivamente implementados en
backend.

\hypertarget{pruebas-unitarias-backend}{%
\subsection{Pruebas unitarias
(backend)}\label{pruebas-unitarias-backend}}

Cobertura de servicios y controladores en módulos críticos (auth,
events, orders, tickets, webhooks, catálogos), con pruebas localizadas
en \texttt{backend/src/**/**.spec.ts}.

\hypertarget{pruebas-de-integraciuxf3n-y-e2e-backend}{%
\subsection{Pruebas de integración y e2e
(backend)}\label{pruebas-de-integraciuxf3n-y-e2e-backend}}

Suites en \texttt{backend/test/} para auth, eventos, órdenes, tickets,
catálogos y concurrencia.

\hypertarget{pruebas-de-concurrenciacarga}{%
\subsection{Pruebas de
concurrencia/carga}\label{pruebas-de-concurrenciacarga}}

Scripts de concurrencia y stress declarados en
\texttt{backend/package.json} (\texttt{test:concurrency},
\texttt{test:load}, \texttt{test:stress}, \texttt{test:soak}).

\hypertarget{pruebas-de-seguridad-funcional}{%
\subsection{Pruebas de seguridad
funcional}\label{pruebas-de-seguridad-funcional}}

Validación de autenticación JWT, permisos por rol y comportamiento de
validación de tickets.

\hypertarget{pruebas-de-acceso-y-usabilidad-frontend}{%
\subsection{Pruebas de acceso y usabilidad
(frontend)}\label{pruebas-de-acceso-y-usabilidad-frontend}}

Frontend con tests unitarios/nuxt/e2e vía Vitest; no obstante, el flujo
completo de compra y scanner QR no está integrado en UI en el estado
actual.
